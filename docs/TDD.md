# t_wada式TDD（厳守）

全てのコード実装においてt_wada式TDD（Kent Beckの定義に基づく5ステップ）を厳守する。
このドキュメントは開発手法の完全な定義であり、外部資料を参照せずにこの内容だけで正確に実践できるようにする。

## レビュー方針

各ステップ（List / Red / Green / Refactor）完了時のレビューはAI自身がセルフレビューで行う。
ユーザーの承認待ちは不要。TDD.md内のチェックリストに従い自己チェックし、問題なければそのまま次のステップに進む。

## 5ステップ

TDDは「Red→Green→Refactor」の3ステップではない。テストリスト作成という設計ステップを含む5ステップである。

### Step 1: List（テストリスト作成）— 設計ステップ

- 実装する機能に対して、テストすべきシナリオのリスト（テストリスト）を書き出す
- テストリストはテストファイル内に `test.todo()` で記述し、TODOリストとして管理する
- 網羅しようと執着せず、備忘録を書くような温度感で書き進める
- 手を動かす過程で気づいたことをリストに随時追加・修正する（リストは生き物）
- テストリストを省略してはならない

#### ▶ Redに進む前に確認（必須）

- [ ] テストリストがSPEC.mdの仕様を網羅しているか
- [ ] 漏れている観点はないか

### Step 2: Red（テストを書き、失敗を確認する）

テストリストから1つだけ選び、実際に具体的で実行可能なテストコードを書く。そしてテストを実行し、失敗することを確認する。

#### Redの条件（最重要ルール）

- テストを実行し、**失敗すること**を確認する
- **意図した理由で失敗していること**を必ず確認する
  - エラーメッセージを読み、なぜ失敗したかを理解する
  - これを怠ると、実装コードが悪いのかテストの書き方が悪いのか原因が掴めなくなる
- 以下は**正しいRed（途中）** である（テストは正しいが、まだGreenに進めない）:
  - 関数やクラスが未定義で失敗（まだ作っていないから当然）
- 以下は**正しいRed（完了）** である（Greenに進んでよい）:
  - 関数はあるが戻り値が期待と違って失敗（まだ実装していないから当然）
- 以下は**正しくないRed**である（テストを修正する必要がある）:
  - テストコードのタイポによる失敗
  - テストコードの構文エラーによる失敗
  - importパスの間違いによる失敗
  - テストの前提条件が間違っている
- **失敗するテストがない状態で本番コードを書いてはならない**

#### Redの段階的な進行

1つのテストに対してRedは段階的に進む。例:

1. テストを書く → 実行 → 「関数が未定義」で失敗 → **Redの途中**（環境エラー）
2. 関数をからっぽで作る → 実行 → 「戻り値がundefined、期待はX」で失敗 → **Redの完了**（仕様と動作の不一致）
3. → Step 3（Green）に進む

#### ▶ Greenに進む前に確認（必須）

- [ ] テストが「仕様と動作の不一致」を理由に失敗しているか（環境エラーではないか）
- [ ] 失敗メッセージを読み、期待値と実際の値を確認したか
- [ ] テストコードにタイポや構文エラーがないか

### Step 3: Green（テストを通す最小限のコードを書く）

Step 2で書いたテスト、および過去に書いた全てのテストが通る最小限のコードを書く。

#### Greenにする3つの戦略

状況に応じて使い分ける:

##### 1. 仮実装（Fake It）

- テストを通すためだけのベタ書きの実装をする
- 例: `add(1, 2)` のテストに対して `return 3` と書く
- まず「テストが通る状態」を作ることが目的
- 自信がないとき、最初の一歩として使う

##### 2. 三角測量（Triangulation）

- テストケースを増やして、仮実装では通らない状況を作り、一般化を強制する
- 例: `add(1, 2)` に加えて `add(3, 4)` のテストを追加 → `return 3` では通らないので一般化が必要になる
- 正しい一般化の方法がわからないときに使う

##### 3. 明白な実装（Obvious Implementation）

- 答えが明らかなとき、仮実装を経由せずに正しい実装を直接書く
- 例: `add(a, b)` に対して最初から `return a + b` と書く
- 自信があるときだけ使う。テストが通らなかったら仮実装や三角測量に戻る

#### ▶ Refactorに進む前に確認（必須）

- [ ] 全テスト（今回追加分＋既存）が通っているか
- [ ] 最小限の実装になっているか（過剰な実装をしていないか）

### Step 4: Refactor（リファクタリング）

全てのテストが通っている状態で、コードの設計を改善する。

- テストコードと本番コードの両方がリファクタリング対象
- リファクタリング中もテストを頻繁に実行し、常に全テストが通ることを確認する
- テストが失敗したらすぐにテストが通っていた状態に戻る
- 重複の除去、命名の改善、構造の整理などを行う
- リファクタリングが不要なら飛ばしてよい

#### ▶ 次のサイクルに進む前に確認（必須）

- [ ] 重複の除去・命名の改善ができているか（または不要と判断したか）
- [ ] 全テストが通っているか
- [ ] UI層の場合: Reactの設計原則（後述）を確認したか

### Step 5: 繰り返し

Step 2に戻り、テストリストから次の1つを選ぶ。テストリストが空になるまで繰り返す。
途中で新たなテスト項目に気づいたらリストに追加する。

## React設計チェックリスト（UI層）

Step 4のRefactor後にUI層のコードを確認する際、以下をチェックする:

- 派生値に `useState` を使っていないか（`useMemo` または計算プロパティで十分な値を state にしていないか）
- `useEffect` 内で `setState` を呼んでいないか（状態の同期は `useMemo` や `useReducer` で行う）
- `useEffect` は外部との連携（DOM操作、タイマー、外部API）のみに使っているか
- イベントハンドラで直接 `setState` できる箇所に冗長な `useEffect` を書いていないか
- `useCallback` / `useMemo` を過剰に使っていないか（パフォーマンス問題が発生してから最適化する）
- コンポーネントが単一責任になっているか（表示と状態管理が混在していないか）

## 歩幅の調整

- TDDの「歩幅」（1サイクルで進む量）は常に調整する
- 自信があるときは歩幅を広げてよい（明白な実装を使う）
- 不安なとき、テストが予想外に失敗したときは歩幅を狭める（仮実装、三角測量に戻る）
- 大規模な機能でも、小さなTODOに分割し、1歩ずつ着実に進めることで不安を取り除く

## 一度に1つだけ

- 一度に複数のテストを書かない
- 一度に1つのテストだけに集中する
- そのテストが通ってから次のテストに進む

## テストの書き方

### Arrange/Act/Assert（AAA）パターン

各テストは以下の3段階で構成する:

1. **Arrange（準備）** — テスト対象の状態をセットアップする
2. **Act（実行）** — テスト対象の操作を1つだけ実行する
3. **Assert（検証）** — 結果が期待通りか検証する

```typescript
test('訪問状態を追加すると配列に追加される', () => {
  // Arrange
  const data = createDefaultSaveData();

  // Act
  const result = addAreaStatus(data, { name: '宿泊', color: '#FF8A65' });

  // Assert
  expect(result.areaStatusModels).toHaveLength(4);
});
```

### テストの独立性

- 各テストは他のテストに依存してはならない
- テストの実行順序に依存してはならない
- 各テストが独自のセットアップを持つこと（共有の状態を変更しない）

### テストの命名

- テスト名は「何をテストしているか」が日本語で明確にわかるようにする
- 例: `'初期データには訪問状態が3件ある'`、`'訪問状態を削除すると該当ユーザーの記録も削除される'`

### テストが仕様である

- テストコードはそのまま仕様書として機能する
- テストを読めば、そのモジュールが何をするか理解できるようにする
- SPEC.mdの仕様はテストとして表現される

## テストリストとSPEC.mdの関係

- テストリスト（Step 1）はSPEC.mdの仕様から導出する
- SPEC.mdの各項目を「テスト可能な具体的シナリオ」に分解してリスト化する
- 例: SPEC.mdに「訪問状態を削除したとき該当記録も削除」とあれば → テスト「訪問状態を削除すると対応するMapDataModelが削除される」

## Vitest と Playwright の使い分け

- **Vitest（単体テスト）**: データ操作ロジック、状態管理、ユーティリティ関数など純粋なTypeScriptコード
- **Playwright（E2Eテスト）**: 画面遷移、ユーザー操作（クリック、入力）、表示の確認
- コアロジックは必ずUIから分離し、Vitestでテストできるようにする
